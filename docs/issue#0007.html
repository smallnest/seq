Design Decisions

Decision "Last" operations (FindLast/FindLastIndex/Last) scan the full sequence rather than materialize-and-index

Ambiguity: FR-4 marks FindLast/FindLastIndex as "内部物化", implying collect-then-scan. Last is not marked.

Choice: FindLast/FindLastIndex/Last stream once, keeping only the last match (or last element) in a scalar, never materializing the whole sequence.

Rationale: A single streaming pass with O(1) extra space finds the last match/element just as correctly as materialize-then-index, without the allocation. The "内部物化" note in the PRD describes the conceptual need to see the whole sequence (you can't know the last until the end), which streaming satisfies. This is strictly better than collecting for large/infinite-friendly sources.

Decision MaxBy/MinBy take a less func(a,b T) bool comparator (not cmp-int)

Ambiguity: FR-4 signs MaxBy(less func(T, T) bool), consistent with SortBy's bool form — but stdlib slices.MaxFunc uses cmp-int.

Choice: Keep the PRD's bool signature: MaxBy(less func(a, b T) bool) treats less(best, v) as "v is greater".

Rationale: Matches the PRD exactly and is consistent with SortBy (issue #6). The bool form is more ergonomic for callers (func(a,b int) bool { return a < b }) than crafting −1/0/+1. Tested with string comparator.

Decision Nth treats negative index as out-of-range (returns (zero,false))

Ambiguity: FR-4 signs Nth(n int) (T, bool) without defining n < 0.

Choice: n < 0 returns (zero, false); n ≥ length returns (zero, false); otherwise the nth element.

Rationale: There is no meaningful element at a negative index; returning not-found is total and panic-free, matching the library's no-panic posture. Tested.

Deviations

None in signature. All 31 FR-4 methods implemented with the exact signatures specified, including the generic-method ones (Fold[U], GroupCount[K], GroupBy[K], KeyBy[K], MaxByKey/MinByKey[K], SumBy/MeanBy[U]). The only deviation is the known go1.27rc1 gofmt parser false-positive on the generic-method lines in terminal_aggregate.go (same as issue #6) — build/vet/test all pass; formatting verified by eye. No instantiation-cycle issue here because none of these methods instantiate Seq[Seq[T]].

Tradeoffs

Tradeoff Two-file split: basic terminals vs aggregate/generic terminals

Alternatives: (A) all 31 in one file; (B) split terminal.go (no method type params) vs terminal_aggregate.go (Fold/GroupBy/MaxByKey/SumBy etc. with method type params).

Pros/cons: (A) single import block; (B) isolates the Go-1.27-generic-method surface, so the gofmt false-positive localizes to one file and the "by key" / "by projection" aggregation family reads as a unit.

Why chosen: (B). It localizes the rc1 tooling quirk to one file (easier to revisit when 1.27 stable lands) and groups the conceptually-related aggregation methods. Consistent with the numeric.go/comparable.go split in #3.

Tradeoff Join uses strings.Builder (polished during review)

Alternatives: (A) []byte accumulator + final string(b); (B) strings.Builder.

Pros/cons: (B) is the idiomatic, escape-analyzed choice; (A) is marginally more direct but less idiomatic.

Why chosen: (B), applied during review for consistency with JoinStrings (#3, same polish). Functionally identical; tested.
