Implementation Notes

Issue #10 — 约束型子类型与链式恢复 (SeqComparable/SeqOrdered/SeqNumeric) — 2026-06-24

Design Decisions

Decision "Inheritance" is realized by re-exposing each constrained method on every stronger subtype, routing through the downgrade chain

Ambiguity: FR-8 says SeqNumeric "继承 ordered 的全部" and SeqOrdered "继承 comparable 的全部" — but Go defined types have no inheritance; only the explicitly-declared method set is available.

Choice: Re-declare each inherited method on the stronger subtype, returning the same subtype (e.g. SeqNumeric.Distinct() SeqNumeric, SeqOrdered.Max(), SeqNumeric.Sort()). Each body routes through the downgrade to the existing free function: SeqNumeric.Distinct → Distinct(s.Ordered().Comparable().Seq()), wrapped back as SeqNumeric[T].

Rationale: This is the only way the headline chain Numbers(...).Distinct().Sum() compiles — Distinct must be callable directly on SeqNumeric, and it must return SeqNumeric (not SeqComparable) or the subsequent .Sum() would be unavailable. The design doc's "继承" is aspirational; in Go it is spelled out as explicit re-exposure. The two intermediate downgrades (Ordered().Comparable()) are zero-cost defined-type conversions, so there's no runtime cost to the indirection. This is the documented "API面成本" the design doc accepts.

Decision Map/FlatMap are intentionally NOT on the subtypes; Seq() drops back to bare Seq first

Ambiguity: FR-8d says changing-T methods "不在子类型上提供，需先 Seq() 退回裸序列" — confirm this is enforced rather than accidentally omitted.

Choice: Only T-preserving intermediates (Filter/Reject/Take/Drop/TakeWhile/DropWhile/Peek) are re-exposed on the subtypes. Map/FlatMap (which need a method-level U that escapes the subtype) are absent; the documented escape is ...Comparable().Seq().Map(...).

Rationale: A subtype method returning Seq[U] would lose the constraint (U isn't constrained), so it correctly belongs on bare Seq[T]. TestSubtypeMapDropsToBareSeq pins the escape path. This matches FR-8d exactly.

Decision Each subtype also gets a Collect() terminal for convenience

Ambiguity: FR-8 lists constrained methods and re-exposed intermediates but doesn't mention a materializing terminal on the subtypes.

Choice: Added Collect() on all three subtypes (routing through the downgrade to Seq.Collect).

Rationale: Without it, every subtype pipeline must end in .Ordered().Comparable().Seq().Collect() or a constrained terminal — excessively verbose for the common "just materialize" case. Collect doesn't change T and doesn't need a constraint, so it's a harmless convenience that keeps terminals ergonomic. Tested on all three.

Deviations

None in signature. All FR-8 elements implemented as specified: three subtypes (defined types over iter.Seq[T] with the right constraints), three entry free functions (Comparable/Ordered/Numbers), three downgrade methods (Ordered/Comparable/Seq), re-exposed T-preserving intermediates returning the same subtype, and the constrained methods on each. The only deviation is the known go1.27rc1 gofmt parser false-positive — but note subtypes.go itself is gofmt-clean here (the generic-method receiver forms it uses parse fine in this file); the flagged file was the test, and that was a genuine trailing-comment alignment fix that gofmt -w resolved. Build/vet/test all pass.

Tradeoffs

Tradeoff Hand-written method repetition across 3 subtypes vs codegen

Alternatives: (A) hand-write each inherited method on each subtype (~30 methods, repetitive); (B) generate with go:generate/stringer-style tooling; (C) embed a shared struct.

Pros/cons: (A) is verbose but trivially auditable and compiles directly; (B) reduces repetition but adds a generation step and build complexity; (C) is impossible — Go can't embed methods that reference the outer defined type's type parameter across a defined-type-over-func boundary.

Why chosen: (A). The design doc explicitly accepts this "API面成本" as the price of restoring the chain. Hand-written is clear, matches the PRD's per-subtype method lists one-to-one, and the bodies are one-liners delegating to existing free functions. Codegen can be revisited if the surface grows.

Open Questions

Question Should the subtypes re-expose MORE terminals (Count, Find, Any, All, ForEach) or just Collect?

Assumption: Only Collect plus the FR-8-listed constrained terminals (Sum/Max/Distinct/...) and the T-preserving intermediates. The Seq-style terminals (Count/Find/Any/All) are reachable via .Seq() but not directly on the subtype.

Verify: If downstream frequently wants Numbers(xs).Filter(...).Count() without dropping to Seq, adding Count/Find/Any/All as convenience methods would help. Confirm whether to keep the subtype surface minimal (current) or mirror Seq's full terminal set. Minimal is safer for v1; can add later without breaking.

Question Is the double-downgrade routing (Ordered().Comparable().Seq()) the intended implementation, or should constrained ops reach the underlying iter.Seq directly?

Assumption: Route through the downgrade chain — it's zero-cost (defined-type conversions) and keeps a single code path (all constrained ops go through the free functions).

Verify: An alternative is Distinct(Seq[T](iter.Seq[T](s))) directly, skipping the named downgrades. Functionally identical and equally zero-cost. The downgrade routing was chosen for readability (mirrors the design doc's Numeric → Ordered → comparable narrative); confirm this style is preferred, or flatten to direct conversion in a polish pass.
