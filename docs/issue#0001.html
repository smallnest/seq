Implementation Notes

Issue #1 — 定义核心类型与约束 (Seq/Seq2/Pair/Tuple3/Tuple4/Numeric) — 2026-06-24

Design Decisions

Decision Numeric constraint uses the ~ approximation operator across all integer and floating-point widths

Ambiguity: The PRD (FR-1) says "Numeric 约束（整型 + 浮点）定义" without enumerating which widths or whether defined types (e.g. type MyInt int) should satisfy it.

Choice: Defined Numeric as an interface union of ~int | ~int8 | ~int16 | ~int32 | ~int64 | ~uint | ~uint8 | ~uint16 | ~uint32 | ~uint64 | ~uintptr | ~float32 | ~float64.

Rationale: The ~ token lets defined types whose underlying type is a numeric kind satisfy the constraint — a near-universal expectation (this is what golang.org/x/exp/constraints and cmp.Ordered do). Omitting ~ would force users to cast their domain types to bare builtins. Including every width (int8..int64, uint..uintptr) matches constraints.Integer ∪ constraints.Float and avoids surprising a user whose element is, say, int32. A test pins this behavior with type myInt int.

Decision Tuple field names are First/Second/Third/Fourth, not V1/V2/... or Field1/...

Ambiguity: FR-1 mandates Tuple3/Tuple4 exist but specifies no field names.

Choice: Ordinal English names: First, Second, Third, Fourth.

Rationale: Go has no tuple literal/destructuring, so fields are accessed by name. Ordinal words read more naturally than V1/Field1 and align with how the design doc discusses positions. Pair keeps the established Left/Right per the PRD's explicit struct { Left A; Right B }.

Deviations

None — implementation followed the spec as written. All types defined exactly as FR-1 mandates: Seq[T any] and Seq2[K,V any] as defined types over iter.Seq/iter.Seq2; Pair[A,B], Tuple3, Tuple4 structs; Numeric constraint. go build ./..., go vet ./..., go test ./..., and gofmt -l all clean.

Tradeoffs

Tradeoff Defined type over iter.Seq[T] vs struct wrapper struct{ it iter.Seq[T] }

Alternatives: (A) defined type type Seq[T any] iter.Seq[T]; (B) struct wrapper holding the iterator.

Pros/cons: (A) gives zero-cost interconversion with iter.Seq[T] and seamless use with slices.Collect/maps.Keys, but its methods cannot add constraints to T. (B) could attach arbitrary state/fields, but every standard-library handoff would need an explicit unwrap.

Why chosen: This is the central decision of the whole library — already settled in the design doc ("为什么是定义类型而不是 struct 包装"). Zero-cost interconversion with the iterator ecosystem is worth more than struct extensibility. Tests assert both directions of the conversion compile and work.

Open Questions

Question Should Numeric include complex64/complex128?

Assumption: No — complex numbers are omitted. Sum/Product/Mean over complexes is a rare need, and cmp.Ordered (which SeqOrdered builds on) excludes them, so including complexes would make the Numeric ⊂ Ordered downgrade invariant (FR-8c) false.

Verify: If a downstream use case needs complex aggregation, revisit whether to add a separate ComplexNumeric rather than widening Numeric (which would break the Ordered downgrade).
