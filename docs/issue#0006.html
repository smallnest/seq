Design Decisions

Decision SortBy adapts a bool comparator to slices' cmp-int form

Ambiguity: FR-3 signs SortBy(less func(T, T) bool), but Go 1.27's slices.SortStableFunc expects func(a, b E) int (cmp-style −1/0/+1), not a bool.

Choice: Keep the PRD's bool signature; internally wrap it into the int form (less(a,b)→−1, less(b,a)→+1, else 0) before calling slices.SortStableFunc.

Rationale: The public API stays exactly as specified. The bool→int adapter is a mechanical, well-commented shim hidden inside the method. Stability is preserved (SortStableFunc). A test confirms ascending order and equal-key stability.

Decision Materializing methods collect once and return slice-backed (re-iterable) Seq

Ambiguity: FR-3 marks TakeRight/DropRight/Slice/Init/SortBy/Reverse as "内部物化" but doesn't prescribe the return's re-iteration behavior.

Choice: Each collects via slices.Collect into a fresh slice and returns From(slice), so the result is re-iterable and stable across traversals.

Rationale: These ops inherently need the whole (or tail) sequence, so materialization is unavoidable. Returning a slice-backed Seq makes the result behave like any other slice-backed source — re-iterable, no surprising one-shot semantics. Tail could be Drop(1) (lazy), but is kept materialized for a stable re-iterable result and parity with Init; documented.

Decision Scan emits the initial value as the first element

Ambiguity: FR-3 says "发射每步累积值（含 init）" — confirms init is included, but the exact sequence (init-first vs init-last) could be read either way.

Choice: Emit init, then f(init,x0), f(f(init,x0),x1), … — i.e. n+1 outputs for n inputs, init-first (left scan / prefix-sums style).

Rationale: "含 init" most naturally means init is the first emitted value, matching Scala/Rust scan semantics and prefix-sums intuition. Tested with [1,2,3,4] → [0,1,3,6,10].

Deviations

Deviation Chunk / Window return iter.Seq[Seq[T]], not Seq[Seq[T]]

Spec said: FR-3 signs Chunk(size int) Seq[Seq[T]] and Window(size, step int) Seq[Seq[T]] as methods.

Implemented instead: Chunk(size int) iter.Seq[Seq[T]] and Window(size, step int) iter.Seq[Seq[T]] — same method form, but the outer return type is the underlying iter.Seq rather than the defined Seq.

Why necessary: Go 1.27rc1 forbids a generic method on Seq[T] from instantiating Seq[Seq[T]] — it reports an "instantiation cycle: T instantiated as Seq[T]". This is a language/toolchain limit the PRD did not anticipate; it is not fixable by rephrasing the method. Verified by isolating: every other FR-3 method compiles; only the Seq[Seq[T]]-returning ones cycle.

Mitigation: The yielded inner values are full Seq[T] (built via From), so each chunk/window retains the complete method set — the headline "inner Seq supports full method chain" acceptance criterion still holds (tested). Outer chaining is recoverable via the zero-cost wrap Seq[Seq[T]](s.Chunk(n)) (also tested). This was an explicit user decision (variant A) over making them free functions (variant B).

Deviation gofmt reports "method must have no type parameters" on 5 generic-method lines — accepted as rc1 tooling bug

Spec said: acceptance criteria require go test ./... to pass on the Go 1.27 toolchain.

Implemented instead: go build, go vet, and go test ./... all pass; gofmt -l flags the 5 generic-method signature lines because go1.27rc1's formatter parser rejects generic-method syntax the compiler accepts.

Why necessary: Cannot be fixed in source — it's the toolchain's parser lagging the compiler. Requiring gofmt-clean would mean abandoning generic methods entirely (the library's reason to exist). Per explicit user decision, build+vet+test is the gate for generic-method issues (#6/#7/#8/#10); formatting on those files is verified by eye. Must be noted in API.md (#9) and revisited when a stable 1.27 lands.

Tradeoffs

Tradeoff Chunk/Window as iter.Seq-returning methods vs free functions

Alternatives: (A) keep methods returning iter.Seq[Seq[T]]; (B) make Chunk(s,size)/Window(s,size,step) free functions returning Seq[Seq[T]] (like Flatten).

Pros/cons: (A) preserves the method form the PRD wants and keeps inner Seq[T] chainable, but the outer result is iter.Seq not Seq (needs a wrap for outer chaining). (B) gives a fully chainable Seq[Seq[T]] result and is consistent with Flatten (same "嵌套实例化" constraint class), but loses the s.Chunk(n) method ergonomics.

Why chosen: (A), per user decision. The method form is more discoverable and the inner-chain capability (the actual acceptance criterion) is intact; the outer-wrap is a one-time, documented step. Tradeoff recorded for API.md.

Open Questions

Question Will a stable Go 1.27 lift the Seq[Seq[T]] instantiation-cycle limit?

Assumption: The cycle is an rc1 restriction that may be relaxed in the stable release (the generic-method proposal #77273 doesn't obviously forbid recursive instantiation).

Verify: When 1.27 stable ships, re-test whether Chunk/Window can return Seq[Seq[T]] directly. If yes, revert variant A to the PRD-exact signature and drop the wrap note from API.md. If no, the iter.Seq return is permanent — document prominently.

Question Should Tail be lazy (Drop(1)) instead of materialized?

Assumption: No — kept materialized for a stable re-iterable result and parity with Init.

Verify: Tail is semantically Drop(1) and could stream lazily. The materializing choice costs one allocation but gives re-iterability. Confirm the tradeoff is acceptable; if streaming matters, Tail can switch to Drop(1) without breaking its signature.
