Issue #4 — 多序列/嵌套自由函数 (Zip/ZipWith/ZipMap/Flatten/Concat...) — 2026-06-24

Design Decisions

Decision All Zip* functions drive inputs with iter.Pull and stop at the first exhausted input

Ambiguity: FR-6 says "短者止" for Zip but does not prescribe a mechanism, and Zip3/Zip4/ZipWith inherit the rule by analogy.

Choice: Pull each input with iter.Pull, advance all in lockstep, and return as soon as any one input reports ok == false. Every stop is deferred so partially-consumed iterators are always released.

Rationale: iter.Pull is the only way to advance multiple iterators in lockstep (range loops can't be interleaved). Deferring all stops — including on the early !yield path — prevents iterator leaks on every exit, which matters for channel-backed or resource-bearing sources. Returning on first exhaustion is the natural "shortest wins" semantics and short-circuits the rest.

Decision ZipMap resolves duplicate keys last-write-wins; empty input yields a non-nil empty map

Ambiguity: FR-6 signs ZipMap[K comparable, V any](keys Seq[K], vals Seq[V]) map[K]V as "配成 map" without specifying duplicate-key behavior.

Choice: Iterate in order, assigning out[k] = v; later pairs overwrite earlier. Empty input returns an empty, allocated (non-nil) map.

Rationale: Last-write-wins matches lodash zipObject and Go map-assignment semantics. A non-nil empty map avoids nil-map-assignment panics downstream and is consistent with ToSet/CountValues from issue #3.

Decision Unzip materializes the source once, serving both result Seqs from shared buffers

Ambiguity: FR-6 says Unzip(s Seq2[A,B]) (Seq[A], Seq[B]) but a Seq2 may be one-shot; the two returned Seqs must each be independently iterable.

Choice: Drain the source into two slices on first use of either returned Seq (guarded by a drained flag), then range over the buffers for each side.

Rationale: The source may be a channel-backed or otherwise one-shot Seq2; both result Seqs must see the full, identical element set and be re-iterable. Materializing once (not per side) avoids draining the source twice, which would be wrong for one-shot sources and wasteful otherwise. The tradeoff — one full allocation — is unavoidable given the two-independent-iterable contract.

Decision Interleave round-robins, skipping exhausted sequences until all are drained

Ambiguity: FR-6 signs Interleave[T any](seqs ...Seq[T]) Seq[T] as "轮流交错取元素" without defining uneven-length behavior.

Choice: One element per sequence per round, in order; a sequence that returns ok == false is nilled out and skipped in subsequent rounds; the loop ends when no sequence remains active.

Rationale: "轮流交错" most naturally means round-robin. Skipping exhausted sequences (rather than stopping when the first ends) preserves all elements, which is the more useful default — a user wanting "stop at shortest" can compose Zip. The exact order is pinned by a test with uneven lengths.

Deviations

None — all 9 FR-6 functions implemented with the exact signatures specified: Zip/ZipWith/Zip3/Zip4 stop at the shortest, Flatten[T](Seq[Seq[T]]) flattens one level, Concat/Interleave are variadic, Unzip splits Seq2 into two Seqs. Flatten is single-level only, matching the Non-Goals (no arbitrary-depth flatten).

Tradeoffs

Tradeoff Zip3/Zip4 repeat the pull-loop boilerplate vs. a generic combiner

Alternatives: (A) write each arity's pull-loop explicitly; (B) build a generic "pull N iterators, build a tuple" helper.

Pros/cons: (A) repeats ~15 lines per arity but each body is trivially readable; (B) would need a way to assemble tuples generically, which Go's type system can't express without reflection or codegen — and would obscure the stop-on-first-exhaustion logic.

Why chosen: (A). The repetition is structural (arity differs), not behavioral, and Go offers no clean abstraction for it without HKT. Explicit loops keep the defer/stop discipline auditable per function. This matches the design doc's "no HKT / type class" boundary.

Open Questions

Question Should Interleave stop at the shortest sequence instead of draining all?

Assumption: No — it drains all, skipping exhausted sequences (round-robin to completion). This was chosen as the more useful default.

Verify: If a downstream use case expects "stop when any sequence ends" (zip-like interleaving), the current behavior will surprise them. Confirm the drain-all semantics is the intended contract before API.md (#9) freezes it, or document both modes explicitly.
