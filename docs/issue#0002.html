Design Decisions

Decision Range / RangeStep auto-detect direction from start vs end

Ambiguity: FR-2 signs Range(start, end int) Seq[int] and RangeStep(start, end, step int) Seq[int] as "[start, end) 整数序列" without saying what happens when start > end.

Choice: When start > end, the range walks downward toward end (exclusive), with step interpreted as a positive magnitude subtracted each iteration. start == end is empty.

Rationale: A literal "range over [start, end)" with start > end would always be empty, which is rarely what a caller writing Range(5, 0) intends. Two-sided behavior mirrors Rust/Python idioms and makes countdown pipelines ergonomic. The behavior is pinned by tests and documented in the doc comment.

Decision RangeStep with step ≤ 0 yields an empty Seq instead of panicking

Ambiguity: FR-2 does not define a zero/negative step.

Choice: Return an empty Seq for step ≤ 0.

Rationale: Constructors are lazy and must be total — panicking inside a deferred yield closure would surface far from the call site. "Empty input, empty output" is the least surprising default and keeps the library panic-free. Documented in the doc comment.

Decision From captures the backing slice by reference (no copy)

Ambiguity: Whether From([]T) should defensively copy the slice.

Choice: No copy; the Seq ranges over the caller's slice directly, so mutations between iterations are visible. Documented on From.

Rationale: Zero-cost is a core promise (the whole library is built on defined types over iter.Seq). A defensive copy would silently allocate on every construction and betray that promise. Callers needing isolation copy before passing in. This matches how slices.All treats its input.

Deviations

None — all 11 constructors from FR-2 are implemented with the exact signatures specified. The acceptance-criteria examples From([]int{1,2,3}).Collect() and Range(0, 3).Collect() are exercised in tests (via a local collect helper using the zero-cost iter.Seq[T] conversion, since the Collect method lands in issue #7).

Tradeoffs

Tradeoff Testing constructors without depending on the not-yet-built Take/Collect methods

Alternatives: (A) implement #2 now and defer its terminal-driven tests until #6/#7 land; (B) test via slices.Collect(iter.Seq[T](s)) and a local takeN loop today.

Pros/cons: (A) keeps the test file "pure" but ships #2 with no verification that its infinite sources are actually bounded-safe — the riskiest property. (B) duplicates a sliver of what Take/Collect will do, but only as private test helpers, and proves the no-hang invariant immediately.

Why chosen: (B). The dependency graph says #2 depends only on #1; shipping it untested on its headline "infinite sources don't hang" guarantee would be irresponsible. The helpers are file-local (collect, takeN), not public API, so they do not pre-empt #6/#7 or create duplication in the shipped surface.

Open Questions

Question Should RangeStep accept a negative step to mean "descend" explicitly?

Assumption: No — direction is inferred from start vs end, and step is always a positive magnitude. A negative step currently yields empty.

Verify: If downstream expects Python-style range(5, 0, -1) semantics, the descending-with-positive-step choice will surprise them. Confirm the documented convention is acceptable, or revisit before API.md (#9) freezes the contract.
