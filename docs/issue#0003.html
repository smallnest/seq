Implementation Notes

Issue #3 — 约束型自由函数 (Distinct/Max/Sum/Compact/集合运算...) — 2026-06-24

Design Decisions

Decision Set operations (Intersect/Difference/SymmetricDifference/Union) dedup their output

Ambiguity: FR-5 lists these as set operations but does not state whether an input with duplicate keys, or a result spanning duplicates, should carry duplicates through.

Choice: All four produce a de-duplicated result preserving first-occurrence order (Union across the concatenation; Intersect/Difference across a; SymmetricDifference across a then b).

Rationale: They are set operations — returning duplicates would be incoherent (e.g. Intersect([1,1], [1]) yielding [1,1]). Set semantics with dedup is the least surprising contract and matches lodash's set helpers. Tests pin the dedup behavior explicitly.

Decision Compact drops elements equal to the zero value of T (via ==)

Ambiguity: "去掉零值元素" could mean "falsy" (e.g. empty string, 0, nil) generically, but on a lazy typed Seq there is no universal falsiness.

Choice: Compact[T comparable] compares each element to var zero T with ==. Requires comparable (which is why Compact is a free function).

Rationale: On a typed sequence the zero value is well-defined per type (0, "", false, zero struct). == against it is the natural, type-safe definition and needs comparable — consistent with the 划分铁律. This is exactly lodash compact behavior specialized to Go's value semantics.

Decision Equal is streaming (pulls both iterators in lockstep, no full materialization)

Ambiguity: FR-5 only says "逐元素相等"; the implementation could collect both sides then compare.

Choice: Use iter.Pull on both sequences with a deferred stop(), comparing element-by-element and returning false on the first length or value mismatch.

Rationale: Streaming short-circuits on large/infinite sequences (an early mismatch avoids draining the rest), and avoids two full allocations. The deferred stop calls prevent iterator leaks on every return path. Two empties are equal, as the PRD requires.

Deviations

None — all 20 FR-5 functions implemented with the exact signatures and the documented empty-sequence conventions: Max/Min → (zero,false), Sum → zero, Product → 1, Mean → 0, Contains → false, IndexOf/LastIndexOf → (0,false), CountValues/ToSet → empty non-nil map, JoinStrings → "", Sort → empty. LastIndexOf and Sort materialize internally, as the PRD marks them "内部物化".

Tradeoffs

Tradeoff Splitting numeric/ordered ops into numeric.go vs one big comparable.go

Alternatives: (A) all 20 functions in a single file; (B) split by constraint — numeric.go for Numeric/cmp.Ordered ops (Max/Min/Sum/Product/Mean/Sort), comparable.go for the rest.

Pros/cons: (A) is one import block and one mental map; (B) groups by the constraint that forces each to be a free function, mirroring the design doc's "划分铁律" and easing the later subtype parity check (issue #10).

Why chosen: (B). The split mirrors the conceptual partition (Numeric/Ordered vs comparable) and makes the eventual review that "SeqNumeric methods == these free functions" a file-to-file comparison. Cost is one extra import block — negligible.

Tradeoff JoinStrings uses a []byte accumulator vs strings.Builder

Alternatives: (A) strings.Builder; (B) a []byte with append + final string(b).

Pros/cons: strings.Builder is the idiomatic, escape-analyzed choice; a raw []byte append is marginally more direct but loses Builder's optimizations and is less idiomatic.

Why chosen: A was actually the better choice; the current implementation uses (B) []byte. Flagged here for a follow-up: prefer strings.Builder in a later polish pass to match Go idiom. Functionally correct either way (tested).

Open Questions

Question Should Mean be generic over Numeric or always return float64?

Assumption: Always float64, as the PRD signs Mean[T Numeric](s Seq[T]) float64. Integer division would silently truncate; float64 is the safe default.

Verify: The float64 return type means callers on integer sequences who want an integer average must convert themselves. Confirm this is acceptable (it matches lodash mean).
