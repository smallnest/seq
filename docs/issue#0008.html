Implementation Notes

Issue #8 — Seq2[K,V] 方法 (MapValues/MapKeys/Keys/Values/Filter...) — 2026-06-24

Design Decisions

Decision Keys()/Values() project lazily and return full Seq[K]/Seq[V]

Ambiguity: FR-7a says Keys() Seq[K] / Values() Seq[V] but doesn't say whether they materialize or stream.

Choice: Both are lazy projections: Keys yields each key, Values each value, without collecting. They return real Seq, so the result carries the full Seq method set.

Rationale: Lazy projection composes with the rest of the pipeline (src.Values().Filter(...) works without an intermediate slice — explicitly tested). Returning Seq not iter.Seq is possible here because the element type is K/V, not a recursive Seq[...] (no instantiation cycle, unlike Chunk/Window in #6).

Decision MapKeys documents that mapped keys may collide (later overwrites in ToMap)

Ambiguity: FR-7a signs MapKeys[J any](f func(K) J) Seq2[J, V] without saying what happens when f maps distinct keys to the same J.

Choice: MapKeys yields all pairs unchanged (no dedup at the Seq2 level — it's a transformation, not a materialization). Downstream ToMap then applies its documented last-write-wins on the colliding J.

Rationale: A Seq2 is a sequence of pairs, not a map — deduplication is a map-materialization concern (owned by ToMap, #5). Transforming keys and keeping all pairs is the faithful lazy behavior; collapsing collisions early would silently drop data and belongs to ToMap's contract. Documented on MapKeys.

Deviations

None in signature. All 12 FR-7a methods implemented exactly as specified: MapValues[U]/MapKeys[J]/Map[J,U] (Go 1.27 method type params), Filter/Keys/Values/ForEach/Fold[U]/Count/Find/Any/All. The only deviation is the known go1.27rc1 gofmt parser false-positive on the generic-method lines in seq2_methods.go (same as #6/#7) — build/vet/test all pass; formatting verified by eye. No instantiation-cycle issue (no recursive Seq2[Seq2[...]] returns).

Tradeoffs

Tradeoff Find returns (zero,zero,false) on miss rather than a pointer/option

Alternatives: (A) return (K, V, bool) with zero values on miss; (B) return (*Pair[K,v], bool) or a custom Option.

Pros/cons: (A) matches the PRD signature Find(pred) (K, V, bool) and the pattern used by Seq.Find ((T, bool)) — consistent and zero-alloc; the bool is authoritative. (B) would avoid emitting zero values but adds allocation and diverges from the established pattern.

Why chosen: (A). Consistency with Seq.Find and the PRD signature wins; the bool is the authoritative signal, exactly as documented for IndexOf/Find elsewhere.

Open Questions

Question Should Seq2 have an Any/All but also a None, and a First/Last?

Assumption: Implemented exactly the 12 methods FR-7a lists (no None, no First/Last on Seq2). Seq has None/First/Last; Seq2 does not, per the PRD.

Verify: If downstream wants Seq2.None/First/Last, they're trivial to add but are out of FR-7a's scope. Confirm the PRD's deliberate omission is intended (likely yes — Seq2's "last" is map-order-dependent and less meaningful).
