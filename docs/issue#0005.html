Issue #5 — Seq2 配套自由函数 (ToMap/CollectPairs/Entries/Associate) — 2026-06-24

Design Decisions

Decision ToMap resolves duplicate keys last-write-wins and returns a non-nil empty map for empty input

Ambiguity: FR-7b signs ToMap[K comparable, V any](s Seq2[K, V]) map[K]V as "物化为 map" without specifying duplicate-key handling or the empty case.

Choice: Iterate in order assigning out[k] = v (later overwrites earlier); empty input returns an empty, allocated (non-nil) map.

Rationale: Last-write-wins is the natural Go map-assignment semantics and matches ZipMap from issue #4. A non-nil empty map avoids nil-map-assignment panics for downstream callers and is consistent with ToSet/CountValues from #3. This is the same convention now applied uniformly across all materialize-to-map functions.

Decision Entries captures the backing Pair slice by reference (re-iterable)

Ambiguity: Whether Entries([]Pair[K,V]) should copy.

Choice: No copy; the Seq2 ranges over the caller's slice directly, so it is re-iterable.

Rationale: Consistent with From (#2) and the library's zero-cost promise. Re-iterability matters because Entries is the inverse of CollectPairs, and round-tripping a slice through Entries→CollectPairs should reproduce it exactly on each traversal. Documented behavior matches From.

Deviations

None — all 4 FR-7b functions implemented with the exact signatures specified: ToMap (constrains K comparable), CollectPairs, Entries, Associate[T any, K comparable, V any] (projects Seq[T] → Seq2[K,V]). All compile on Go 1.23+ as the PRD notes ("不依赖 Go 1.27"), since they are free functions over iter.Seq2.

Tradeoffs

Tradeoff Associate returns a lazy Seq2 rather than materializing eagerly

Alternatives: (A) lazy Seq2 that re-runs f per element on each iteration; (B) materialize into a []Pair once.

Pros/cons: (A) is zero-allocation until consumed and composes with the rest of the lazy pipeline, but re-runs f if iterated twice; (B) runs f exactly once but allocates and breaks laziness.

Why chosen: (A). The library's whole posture is lazy composition — Associate feeding ToMap or MapValues should not force an intermediate allocation. Re-running a pure projection on re-iteration is acceptable and matches how Map/Filter behave downstream. Callers wanting memoization can CollectPairs the result.

Open Questions

Question Should CollectPairs/Entries use a more semantic name than Pair.Left/Right for KV pairs?

Assumption: No — reuse the existing Pair[K,V] with Left=key, Right=value. The PRD explicitly defines CollectPairs as producing []Pair[K, V].

Verify: For KV semantics Key/Value field names read more naturally than Left/Right, but introducing a separate KV type would fragment the type space and break the round-trip with Entries/Zip. Confirm Pair reuse is acceptable; if KV readability matters, document the Left=key/Right=value convention in API.md (#9) rather than adding a type.
