Design Decisions

Decision Generated signatures by extracting them from the shipped code, not by transcribing the PRD

Ambiguity: FR-9 requires "文档与代码签名一致（可人工核对）" — the PRD itself lists intended signatures, but the shipped code diverged in places (e.g. Chunk/Window return iter.Seq[Seq[T]], SortBy wraps a bool→int adapter).

Choice: Used grep over the merged .go files to enumerate every free function and method (by receiver type), then authored API.md from that ground truth. Spot-checked key signatures (Map, DistinctBy, Chunk, Fold, Numbers) for verbatim parity code↔doc.

Rationale: The PRD is the design intent; the code is the contract. A doc derived from the PRD would silently replicate the PRD's pre-deviation signatures and contradict the shipped API. Deriving from code guarantees the "一致" acceptance criterion holds and makes the deviation (Chunk/Window) surface explicitly rather than being papered over.

Decision Documented the Chunk/Window deviation and the gofmt rc1 false-positive inline in API.md

Ambiguity: FR-9 wants a clean authoritative contract; it doesn't say whether to surface toolchain limitations.

Choice: API.md §1.3 calls out the iter.Seq[Seq[T]] return-type deviation with the wrap workaround; §6 documents the go1.27rc1 gofmt parser false-positive on generic methods.

Rationale: An authoritative contract that hides the one real deviation would mislead anyone implementing against it. Surfacing both — with the exact workaround (Seq[Seq[T]](s.Chunk(n))) and a "revisit on stable 1.27" note — makes the doc honest and actionable. These are the two facts a downstream reader most needs.

Decision Methods tagged intermediate/terminal; free functions tagged with their "can't be a method" reason category

Ambiguity: FR-9 asks for both tags but doesn't prescribe a layout.

Choice: Seq[T] methods split into 1.1 (lazy intermediate), 1.2 (materializing intermediate), 1.3 (the Chunk/Window deviation), 1.4 (terminal). Free-function sections (3.2 constrained, 3.3 multi-seq) carry a "为何不是方法" column with the reason class (约束 T / 多类型参数 / 嵌套实例化).

Rationale: The intermediate/terminal split directly serves the "标注中间或终结" criterion and mirrors how a user reasons about laziness. The reason-class column makes the 划分铁律 mechanically auditable from the doc alone — exactly the PRD's Success Metric ("每个自由函数都能对应到三类判定理由之一").

Deviations

None from FR-9's own spec. API.md is generated exactly as required: method/free-function partition, one-line semantics, intermediate/terminal tags, free-function reason classes, a dedicated subtype section explaining "入口为函数、降级为方法", and signature parity with code. The deviations documented inside API.md (Chunk/Window return type, gofmt rc1 note) are inherited from issues #6/#10 and are reported there — API.md merely records them faithfully rather than deviating from FR-9.

Tradeoffs

Tradeoff Hand-authored Markdown tables vs generated from doc comments

Alternatives: (A) hand-write API.md from the grep'd signatures; (B) generate it from Go doc comments via a tool (e.g. a custom go doc scrape or godoc-to-markdown).

Pros/cons: (A) is full control over layout, tagging, and the reason-class/narrative columns the PRD wants, but must be manually kept in sync; (B) stays auto-in-sync with code but can't easily express the "why not a method" rationale or the intermediate/terminal/materializing taxonomy (those aren't in the code).

Why chosen: (A). The PRD's value-add over raw godoc is precisely the reason-class and intermediate/terminal annotations, which don't exist in code. Hand-authoring with a grep-derived signature base gives both accuracy and the richer taxonomy. Added a §6 note that the doc must be refreshed when signatures change; a future polish pass could add a parity test that greps code signatures and fails the build if API.md drifts.
