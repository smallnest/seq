# Architecture Smell Report

**Project:** github.com/smallnest/seq
**Scope:** Entire project (12 source files, 8 test files, ~2,240 source LOC / ~1,830 test LOC)
**Date:** 2026-06-25
**Analyzer:** smell skill (Ducc)

---

## Executive Summary

`seq` is a generic, lazy sequence/iterator library built on Go 1.27's `iter.Seq` / range-over-func and method-level type parameters (`golang/go#77273`). The codebase is **healthy and well-engineered**. It exhibits a clean, intentional layering (core types → constructors → intermediate ops → terminal ops → constrained subtypes), has **no circular dependencies, no package-level mutable state, no `init()` functions**, and `go vet` passes clean. Every one of the 12 source files has a corresponding test file, with 107 test functions and a healthy ~0.82:1 test-to-source ratio.

This is **not a "smelly" codebase** in the usual sense — there are no Critical issues, no God Objects, no anemic-vs-rich-model confusion, no leaky abstractions, and no coupling hotspots. The yield-protocol (`if !yield(...) { return }`) is handled correctly in every operator, which is the single most common correctness bug in range-over-func libraries and is **absent here**.

The findings that remain are about **hardening and polish**, not structure:
1. **No benchmarks** for a performance-sensitive iterator library (Warning).
2. **No nil-function validation** on higher-order operators — they panic on `nil` callbacks, and this behavior is neither tested nor documented (Warning/Suggestion).
3. **No `Example` functions** for godoc (Suggestion).
4. Heavy but **idiomatic** boilerplate duplication of the yield-loop scaffold (Suggestion — largely unavoidable in Go).
5. A few **documentation precision** nits (e.g. `RangeStep` half-open-interval wording; materialization not flagged in some doc comments).

## Detailed Findings

### Finding #1: No benchmark suite

- **Category:** Testing / Complexity
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Missing performance regression guard
- **Location:** repository-wide (`grep "func Benchmark"` → 0 results)
- **Principle Violated:** "Measure, don't guess" — a lazy-iterator library lives and dies by allocation and per-element overhead
- **Description:** The library makes deliberate performance claims (zero-cost `Seq`↔`iter.Seq` conversion, lazy streaming, single-pass operators). None of these are guarded by benchmarks. Regressions in allocation behavior (e.g. an accidental `slices.Collect` creeping into a streaming operator) would pass silently. The materializing operators (`SortBy`, `Reverse`, `Window`, `TakeRight`, `LastIndexOf`) have real O(n)-memory behavior that benchmarks would document and pin.
- **Recommendation:** Add a `bench_test.go` with `Benchmark`s covering (a) a streaming chain (`From→Filter→Map→Collect`) to track per-element overhead/allocs, (b) the materializing operators to document their memory profile, and (c) the `iter.Pull`-based ops (`Zip`, `Interleave`, `Equal`) which carry pull-iterator overhead. Run with `-benchmem`.

### Finding #2: Higher-order operators panic on nil callbacks (undocumented, untested)

- **Category:** Design / Code
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Unvalidated precondition / undocumented panic contract
- **Location:** `intermediate.go:15` (`Map`), `:55` (`Filter`), `:26` (`FlatMap`), `:151` (`Scan`); `terminal_aggregate.go` (`Fold`, `GroupBy`, etc.); essentially every operator taking a `func` argument
- **Principle Violated:** Principle of least astonishment; explicit contracts
- **Description:** Passing `nil` to e.g. `s.Map(nil)` produces a `nil`-pointer panic only when the sequence is later driven — far from the call site, making it hard to diagnose. This is a defensible design choice (Go's `slices.SortFunc` also panics on `nil`), but here it is **neither documented nor tested**, so it is an implicit contract. Only 4 `nil`-mentioning lines exist across all test files, none asserting callback behavior.
- **Evidence:**
  ```go
  func (s Seq[T]) Map[U any](f func(T) U) Seq[U] {
      return Seq[U](func(yield func(U) bool) {
          for v := range iter.Seq[T](s) {
              if !yield(f(v)) { return }   // panics here, not at the Map call
          }
      })
  }
  ```
- **Recommendation:** Pick one and make it explicit: (a) document "f must be non-nil; passing nil panics on iteration" in the doc comments of higher-order ops, **or** (b) add an early `if f == nil { panic("seq: nil function") }` at construction time so the panic fires at the call site. Either way, add a test asserting the chosen behavior. Option (b) gives a far better failure locus.

### Finding #3: No godoc `Example` functions

- **Category:** Testing / Documentation
- **Severity:** 🔵 Suggestion
- **Location:** repository-wide (`grep "func Example"` → 0 results)
- **Description:** The API is well-documented in prose (doc comments + `API.md`), but there are no `Example` functions, so godoc shows no runnable, compiler-checked usage snippets. For a fluent/chaining library, examples are the single highest-value doc artifact and they double as compile-checked tests of the public surface.
- **Recommendation:** Add `Example` functions for the headline flows: a `From(...).Filter(...).Map(...).Collect()` chain, a `Seq2` map flow, and one constrained-subtype flow (`Comparable(...).Distinct()`). Use `// Output:` blocks so they run under `go test`.

### Finding #4: Idiomatic yield-loop boilerplate duplicated across ~20 operators

- **Category:** Code (DRY)
- **Severity:** 🔵 Suggestion
- **Anti-Pattern:** Duplicated Code (but idiomatic)
- **Location:** `intermediate.go` (Map/Filter/Reject/Take/Drop/TakeWhile/DropWhile/Peek/Intersperse/Concat/DistinctBy/FilterMap/Scan), `comparable.go` (Compact/Without/Union/Intersect/Difference), `seq2_methods.go`
- **Description:** Nearly every operator repeats the scaffold:
  ```go
  return Seq[U](func(yield func(U) bool) {
      for v := range iter.Seq[T](s) {
          /* per-op logic */
          if !yield(...) { return }
      }
  })
  ```
  This is real duplication, but in Go pre-iterators-helpers it is the **idiomatic and most readable** form — extracting it behind a higher-order combinator would obscure control flow, defeat inlining, and is exactly what the library itself exists to provide to *callers*. **Do not over-refactor this.** Flagged only for awareness.
- **Recommendation:** Leave as-is. If anything, ensure new operators are added by copying an existing one verbatim to preserve the uniform yield-handling. (The uniformity is itself a correctness asset — see Finding-positive note below.)

### Finding #5: Documentation precision nits

- **Category:** Naming / Documentation
- **Severity:** 🔵 Suggestion
- **Locations:**
  - `constructors.go:60-64` — `RangeStep` doc says "integers in `[start, end)` advancing by step" but for the descending branch (`start > end`) the interval is effectively `(end, start]` walked downward; the half-open `[start, end)` phrasing is misleading for the descending case.
  - `comparable.go:57` (`LastIndexOf`), `terminal.go` (`Last`/`FindLast`) — these materialize the full sequence; `LastIndexOf` documents it ("materializes the input internally"), but verify the parallel `Last`/`FindLast` doc comments note the same so callers don't assume streaming on infinite sequences.
- **Recommendation:** Tighten the `RangeStep`/`Range` doc wording for the descending case, and ensure every materializing terminal/intermediate op states it materializes (so the "safe on infinite sequences if bounded" contract stays accurate).

### Finding #6: Thin edge-case coverage in spots

- **Category:** Testing
- **Severity:** 🔵 Suggestion
- **Locations:** `numeric.go` (covered mostly transitively via subtype tests), nil-callback paths (Finding #2), single-element/empty boundaries on a few newer ops
- **Description:** Coverage is broad and behavior-focused (good — no test-implementation coupling, no I/O-dependent/flaky tests, and early-termination is explicitly tested at `intermediate_test.go:327` `TestLazinessShortCircuit`). Gaps are narrow: direct tests for `numeric.go` free functions and explicit nil/empty edge assertions on some operators.
- **Recommendation:** Add targeted tests for `numeric.go` (`Max`/`Min`/`Sum`/`Product`/`Mean`/`Sort` on empty, single-element, and all-equal inputs) and the nil-callback contract chosen in Finding #2.

## Note on Complexity (Positive Findings)

The complexity scan looked hard for the usual iterator-library traps and found the library **already does the right thing**:

- **Set/Map membership, not linear scan:** `Distinct` (`comparable.go:16`), `Without` (`:117`), `Intersect`/`Difference`/`SymmetricDifference`, and `DistinctBy` (`intermediate.go:216`) all use `map`-backed O(1) membership — no O(n·m) scans-in-loops. `Contains`/`IndexOf` are single-pass short-circuiting linear scans, which is correct (building a set for a one-shot membership test would be slower).
- **Materialization is isolated and intentional:** Operators that must buffer (`SortBy`, `Reverse`, `Window`, `TakeRight`/`DropRight`/`Slice`/`Init`/`Tail`, `LastIndexOf`, `Unzip`) are physically separated into `intermediate_materializing.go` or clearly documented, and they materialize because their semantics require it — not by accident.
- **`Window` materializes once** (`intermediate.go:276`), then re-scans with slice windows sharing the immutable backing array — O(n) memory, O(n) windows, no per-window re-collection. Correct.
- **Pull iterators used correctly:** `Zip`/`Zip3`/`Zip4`/`ZipWith`/`Interleave`/`Equal` use `iter.Pull` with `defer stop()` for proper cleanup — **no goroutine/channel leaks** (`multiseq.go:14-250`, `comparable.go:79`).
- **No nested-loop O(n²) anti-patterns.** `Interleave`'s nested loop is O(total elements), not pairwise.

## Refactoring Roadmap

### Immediate Actions (This Sprint)
1. Decide and **document/enforce the nil-callback contract** (Finding #2) — fastest, highest-clarity win.
2. Add a small **`bench_test.go`** covering one streaming chain, one materializing op, and one pull-based op with `-benchmem` (Finding #1).

### Short-Term (1–3 Months)
1. Add **`Example` functions** for the headline flows (Finding #3) — compile-checked docs.
2. Tighten **doc comments**: `RangeStep` descending-interval wording; confirm every materializing op states it (Finding #5).
3. Backfill **`numeric.go` direct edge-case tests** and nil-callback tests (Finding #6).

### Long-Term (3–12 Months)
1. Maintain operator uniformity: when adding operators, **copy an existing operator's yield scaffold verbatim** to preserve the correct early-termination handling — the duplication (Finding #4) is intentionally *not* worth abstracting away.
2. As the API stabilizes, consider a **fuzz test** for the materializing/`iter.Pull` operators to harden against pathological sequences.
